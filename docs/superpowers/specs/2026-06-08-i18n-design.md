# i18n Design — Arabic & English Support

**Date:** 2026-06-08
**Status:** Approved

## Summary

Add internationalization to the trading-logs Next.js app supporting English (default) and Arabic. Locale is cookie-based — no URL changes. Arabic activates RTL layout and an Arabic-compatible font.

## Decisions

| Question | Decision |
|---|---|
| Locale strategy | Cookie-based (`NEXT_LOCALE`), no URL prefix |
| Default locale | English (`en`) |
| Switcher placement | Beside the existing button (temporary; full layout TBD) |
| Arabic translations | Real strings added now |
| Library | `react-i18next` |

## Architecture

The locale is stored in a `NEXT_LOCALE` cookie (default: `en`, expiry: 1 year, `SameSite=Lax`). On every render, `app/layout.tsx` reads the cookie via `next/headers` and:

- Sets `lang` on `<html>` (e.g. `lang="ar"`)
- Sets `dir` on `<html>` (`ltr` for English, `rtl` for Arabic)
- Initializes i18next with the correct locale
- Wraps children in `I18nextProvider`

Client components use the `useTranslation()` hook. Server components call a server-side helper that reads the same cookie and returns an initialized i18next instance.

Switching locale: the `LocaleSwitcher` client component calls a server action `setLocale(locale)` that writes the cookie, then calls `router.refresh()` on the client to re-render.

## File Structure

```
messages/
  en.json                       # English strings
  ar.json                       # Arabic strings
lib/
  i18n.ts                       # Shared config: supported locales, namespaces, fallback
  i18n-server.ts                # Server helper: reads cookie, returns i18next instance
components/
  locale-switcher.tsx           # "EN / العربية" toggle, client component
app/
  layout.tsx                    # Modified: reads cookie, sets lang/dir, adds I18nextProvider
  page.tsx                      # Modified: uses useTranslation()
```

No changes to `next.config.ts`.

## Translation Strings

Namespace structure (flat, two namespaces):

```json
{
  "common": {
    "button": "Button",
    "close": "Close"
  },
  "home": {
    "title": "Project ready!",
    "subtitle": "You may now add components and start building.",
    "buttonHint": "We've already added the button component for you.",
    "darkModeHint": "Press d to toggle dark mode"
  },
  "drawer": {
    "title": "Drawer",
    "description": "This is a drawer."
  }
}
```

Arabic file uses the same keys with Arabic values.

## Locale Switcher

- Renders as a ghost button next to the existing button (temporary placement)
- Label shows the *other* language: "العربية" when in English, "English" when in Arabic
- On click: calls `setLocale()` server action → writes cookie → `router.refresh()`

## Font

Current fonts (Inter, Geist) do not cover Arabic glyphs. **Cairo** (Google Fonts) covers both Latin and Arabic in one family and is applied when `dir="rtl"` is active via a CSS variable.

## Out of Scope

- URL-based routing (`/en/`, `/ar/`)
- Additional languages beyond English and Arabic
- Full UI redesign (locale switcher placement is temporary)
- Middleware-based locale detection
